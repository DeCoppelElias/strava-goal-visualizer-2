# TASK-8.3 — DB Statistics Doc + `make stats` Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Give the operator a documented, repeatable way to read usage statistics from PostgreSQL — locally and against the deployed Fly database — plus a `make stats` shortcut for the common subset.

**Architecture:** A single canonical `scripts/stats.sql` holds the curated summary query. `make stats` runs it against the local compose database via `docker compose exec`; production runs the *same file* through `fly postgres connect`. A reference doc (`docs/ops/db-statistics.md`) documents connecting (local + Fly), the full query catalogue, and the limitations (no `last_seen_at`, signed-cookie sessions).

**Tech Stack:** PostgreSQL 16, `psql`, Docker Compose, GNU Make, Fly.io CLI.

**Spec:** `docs/superpowers/specs/2026-06-18-observability-basics-design.md`

**One-commit rule:** Per `CLAUDE.md`, one backlog task = one commit. Build and verify all three files, then make a **single commit** in the final task. Do not commit between Tasks 1–3.

> **Deviation from spec (intentional):** The spec proposed `make stats` running `psql "$(DATABASE_URL)" -f scripts/stats.sql`. That does not work out of the box because (a) the compose `db` service publishes no host port, so host `psql` cannot reach it, and (b) the app's `DATABASE_URL` uses the `postgresql+asyncpg://` driver suffix, which `psql` cannot parse. This plan routes `make stats` through `docker compose exec -T db psql` (works with zero setup, no host `psql`) and documents the Fly one-liner for production. The `scripts/stats.sql` file is shared by both paths (DRY).

## Task 1: Curated stats query (`scripts/stats.sql`)

**Files:**
- Create: `scripts/stats.sql`

- [ ] **Step 1: Create `scripts/stats.sql`**

```sql
-- Curated usage summary for `make stats` and the Fly prod command.
-- All values cast to text so they share one result set. Order is preserved.
SELECT 'Total users'              AS metric, count(*)::text AS value FROM users
UNION ALL
SELECT 'Signups (last 7 days)',   count(*)::text
  FROM users WHERE created_at > now() - interval '7 days'
UNION ALL
SELECT 'Users who have synced',   count(DISTINCT user_id)::text FROM activities
UNION ALL
SELECT 'Users with a goal set',   count(*)::text FROM goals
UNION ALL
SELECT 'Total activities',        count(*)::text FROM activities
UNION ALL
SELECT 'Total distance (km)',     coalesce(round(sum(distance_meters) / 1000, 1), 0)::text
  FROM activities
UNION ALL
SELECT 'Clubs tracked',           count(*)::text FROM clubs;
```

## Connecting

### Local (Docker Compose)

The compose `db` service does not publish a host port, so connect through the
container:

```bash
# Interactive session
docker compose exec db psql -U postgres -d strava_dev

# Run the curated summary file
docker compose exec -T db psql -U postgres -d strava_dev -f - < scripts/stats.sql
```

### Production (Fly.io)

The PostgreSQL port is never exposed publicly. Use the Fly CLI. Replace
`<db-app>` with your Postgres app name (`fly apps list`).

```bash
# Quick interactive session
fly postgres connect -a <db-app>

# Run the curated summary file with no tunnel (recommended quick check)
fly postgres connect -a <db-app> < scripts/stats.sql

# Open a tunnel for a GUI client (e.g. TablePlus, DBeaver) or local psql
fly proxy 5432 -a <db-app>      # leaves localhost:5432 forwarded to the prod DB
# then, in another terminal:
psql postgresql://postgres:<password>@localhost:5432/<db-name>
```

> Cross-reference: `fly logs -a <app>` streams the live plain-text logs (each line
> carries an `X-Request-ID`), useful for correlating anomalies you spot in the
> stats.

### Read-only role (recommended)

Run statistics through a role that cannot mutate data:

```sql
CREATE ROLE stats_reader WITH LOGIN PASSWORD 'choose-a-strong-password';
GRANT CONNECT ON DATABASE <db-name> TO stats_reader;
GRANT USAGE ON SCHEMA public TO stats_reader;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO stats_reader;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO stats_reader;
```

Then connect with `stats_reader` instead of the app's role.

## Query catalogue

### Users

```sql
-- Total registered users
SELECT count(*) FROM users;

-- Signups per week (most recent first)
SELECT date_trunc('week', created_at) AS week, count(*)
FROM users GROUP BY 1 ORDER BY 1 DESC;

-- Most recent signups
SELECT id, display_name, created_at
FROM users ORDER BY created_at DESC LIMIT 20;
```

### Engagement

```sql
-- Users who have synced at least one activity
SELECT count(DISTINCT user_id) FROM activities;

-- Users with a yearly goal set
SELECT count(*) FROM goals;

-- Activities per user (most active first)
SELECT u.id, u.display_name, count(a.id) AS activities
FROM users u LEFT JOIN activities a ON a.user_id = u.id
GROUP BY u.id, u.display_name
ORDER BY activities DESC;
```

### Clubs

```sql
-- Clubs tracked
SELECT count(*) FROM clubs;

-- Members per club (largest first)
SELECT c.id, c.name, count(m.user_id) AS members
FROM clubs c LEFT JOIN club_memberships m ON m.club_id = c.id
GROUP BY c.id, c.name
ORDER BY members DESC;
```

### Content volume

```sql
-- Total activities and combined distance (km)
SELECT count(*) AS activities,
       round(sum(distance_meters) / 1000, 1) AS total_km
FROM activities;

-- Data date range
SELECT min(start_date) AS earliest, max(start_date) AS latest
FROM activities;
```

> All stored activities are runs (`sport_type = 'Run'` is enforced at ingest),
> so no sport filter is needed.
```
