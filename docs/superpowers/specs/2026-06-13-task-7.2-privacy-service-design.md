# TASK-7.2 ‚Äî PrivacyService: User Data Deletion

_Date: 2026-06-13_

## Context

TASK-7.2 introduces the `privacy/` domain and a `PrivacyService` class. The deletion method is the first method on this service; TASK-7.3 will add `export_user_data` to the same class. The service is consumed by two future endpoints: `POST /privacy/delete` (TASK-7.4) and `POST /strava/deauth` (TASK-7.5).

## PrivacyService

Single public method for this task:

```python
async def delete_user_data(
    self, db: AsyncSession, user_id: int, reason: DeletionReason
) -> None
```

**Parameters:**
- `user_id` ‚Äî internal DB `users.id` (not `strava_athlete_id`). Both callers have this: the privacy delete endpoint gets it from `get_current_user`; the deauth endpoint looks it up by `strava_athlete_id` before calling the service.
- `reason` ‚Äî `DeletionReason` enum (`USER_INITIATED` or `STRAVA_DEAUTH`).

**Execution order** (respects FK constraints ‚Äî children before parent):

1. Load `strava_athlete_id` from the `users` row. If no user exists for `user_id`, return immediately ‚Äî no audit event, no error (idempotent).
2. Bulk DELETE child tables using `sqlalchemy.delete()` statements ‚Äî no ORM row loading:
   - `activities WHERE user_id = ?`
   - `club_memberships WHERE user_id = ?`
   - `oauth_credentials WHERE user_id = ?`
   - `sync_state WHERE user_id = ?`
   - `goals WHERE user_id = ?`
3. DELETE the `users` row.
4. INSERT `deletion_events` row: `user_id=strava_athlete_id`, `reason=reason.value`, `deleted_at=now()`.
5. Log `"User data deleted"` at INFO level ‚Äî no IDs or PII in the log message.

**Transaction ownership:** The method does not call `db.commit()`. The caller (router) owns the transaction, consistent with the project-wide convention documented in `docs/design.md`.

## Testing

Integration tests using the `db` fixture (real PostgreSQL via testcontainers):

1. **Full deletion:** Seed a user with activities, goal, sync state, oauth credentials, and club membership ‚Üí call `delete_user_data` ‚Üí assert all child rows gone, `users` row gone, one `deletion_events` entry with correct `strava_athlete_id` and `reason`.
2. **Idempotency:** Call `delete_user_data` a second time with the same `user_id` ‚Üí no error, no duplicate audit event.
3. **Reason values:** Verify `USER_INITIATED` and `STRAVA_DEAUTH` both write the correct string to the `reason` column.
