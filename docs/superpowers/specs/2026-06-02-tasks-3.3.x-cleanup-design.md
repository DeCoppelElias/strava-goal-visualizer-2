## Overview

Six small ad-hoc cleanup tasks that harden the auth/sync layer before the sync endpoint (TASK-3.4) is built. They address timezone safety, error handling, exception consolidation, pagination, and transaction ownership.

---

## TASK-3.3.1 — Fix timezone-naive `token_expires_at`

**Problem:** `OAuthCredentials.token_expires_at` and `OAuthStateToken.expires_at` are stored as timezone-naive `DateTime` columns. `ensure_fresh_token` compares them with `datetime.now(UTC)` (timezone-aware), which raises `TypeError` in production.

**Changes:**
- `backend/shared/models.py`:
  - `OAuthCredentials.token_expires_at`: `mapped_column()` → `mapped_column(DateTime(timezone=True))`
  - `OAuthStateToken.expires_at`: same fix
- New Alembic migration:
  - `ALTER COLUMN token_expires_at TYPE TIMESTAMPTZ USING token_expires_at AT TIME ZONE 'UTC'` on `oauth_credentials`
  - Same on `oauth_state_tokens.expires_at`

**Testability:** `uv run pytest` passes. `ensure_fresh_token` called with a fresh-from-DB row does not raise `TypeError`.

## TASK-3.3.2 — Wrap httpx network errors in domain exceptions

**Problem:** `httpx.ConnectError`, `httpx.TimeoutException`, and other `httpx.RequestError` subclasses propagate as raw httpx exceptions instead of domain exceptions. Callers have no clean way to catch network failures.

**Changes:**
- `ensure_fresh_token` (`auth/strava_oauth_service.py`): outer `try/except httpx.RequestError` wrapping the entire `async with httpx.AsyncClient()` block → re-raise as `TokenRefreshError`
- `fetch_activities` (`sync/strava_client.py`): same outer guard → re-raise as `StravaAPIError`
- Tests: mock `httpx.ConnectError` → assert correct domain exception raised in each

**Note:** The inner `HTTPStatusError` handlers remain unchanged; the outer guard catches only network-level errors that bypass them.

## TASK-3.3.3 — Guard malformed Strava token responses

**Problem:** If Strava returns an unexpected 200 OK body in `ensure_fresh_token`, the key accesses (`access_token`, `refresh_token`, `expires_at`) raise an unhandled `KeyError`.

**Changes:**
- `ensure_fresh_token`: wrap the three key accesses in `try/except KeyError` → `TokenRefreshError("Strava token response missing expected fields")`
- Replace `return token_data["access_token"]  # type: ignore[no-any-return]` with `return cast(str, token_data["access_token"])`
- Test: mock 200 response with `{}` body → `TokenRefreshError` raised

## TASK-3.3.4 — Consolidate `StravaAPIError` and `StravaUnauthorizedError`

**Problem:** `StravaAPIError` is defined independently in both `auth/exceptions.py` and `sync/exceptions.py`. A single `except StravaAPIError` cannot catch errors from both. `StravaUnauthorizedError` also lives only in `sync/exceptions.py`.

**Changes:**
- New `backend/shared/exceptions.py`:
  ```python
  class StravaAPIError(Exception):
      pass

  class StravaUnauthorizedError(Exception):
      pass
  ```
- `auth/exceptions.py`: remove local `StravaAPIError`; import from `backend.shared.exceptions`
- `sync/exceptions.py`: remove both local classes; import both from `backend.shared.exceptions`
- All import paths updated across the codebase; no behavior change

**Testability:** `from backend.auth.exceptions import StravaAPIError` and `from backend.sync.exceptions import StravaAPIError` resolve to the same class. All existing tests pass.

## TASK-3.3.5 — `fetch_all_activities` pagination helper

**Problem:** `fetch_activities` fetches a single page (max 200 results). Users with more than 200 activities lose data silently.

**Changes:**
- New function in `backend/sync/strava_client.py`:
  ```python
  async def fetch_all_activities(
      access_token: str,
      *,
      after: int | None = None,
  ) -> list[dict[str, Any]]:
  ```
  Loops `fetch_activities` with incrementing page numbers, accumulating results, until a page returns fewer than `per_page` items (i.e., the last page).

**Tests (respx):**
- 2 full pages (200 each) + 1 empty page → correct combined list, 3 requests made
- 0 activities → empty list, 1 request made
- Exactly 200 activities (one full page, then empty) → 2 requests made

## TASK-3.3.6 — Transaction ownership: remove `db.commit()` from `ensure_fresh_token`

**Problem:** `ensure_fresh_token` calls `await db.commit()` after updating credential fields. When called from the sync endpoint (TASK-3.4), which has its own pending ORM state, this mid-operation commit would flush incomplete data prematurely — working against the session lifecycle that `get_db` is designed to manage.

**Convention (aligns with canonical FastAPI/SQLAlchemy pattern):**
> Service methods are transaction-participants, not transaction-owners. They mutate ORM objects; the router or outermost calling service calls `db.commit()`.

This is already how `_upsert_credentials` behaves. `ensure_fresh_token` is the exception to align.

**Changes:**
- `ensure_fresh_token`: remove `await db.commit()`; method only mutates `creds.*` fields
- `docs/design.md`: new subsection under service conventions:
  > **Transaction ownership:** Service methods mutate ORM objects but do not commit. The router or calling service owns `db.commit()`.
- Tests: refresh path updated to assert `db.commit` is NOT called

**Scope note:** `process_callback` and `revoke_tokens` are self-contained operations with no composing callers; they retain their own commits for now.
