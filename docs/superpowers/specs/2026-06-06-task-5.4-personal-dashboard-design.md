# TASK-5.4 — Personal Goal Dashboard Page

_Date: 2026-06-06_

---

## Overview

Extend the existing `DashboardPage.tsx` with goal progress content below the sync card. The dashboard fetches `GET /dashboard/personal` on mount and re-fetches after a successful sync or goal save. The backend is augmented to include a daily cumulative distance series for the pace chart.

---

## Backend Changes

### New schema: `DailyDistancePoint`

```python
class DailyDistancePoint(BaseModel):
    date: str          # YYYY-MM-DD
    cumulative_km: float
```

### Updated: `PersonalDashboardResponse`

Add one field:

```python
daily_series: list[DailyDistancePoint]
```

Only dates with at least one run are included. An empty list is valid (user synced, zero runs this year).

### Updated: `DashboardService.get_personal_dashboard()`

One additional query: fetch current-year run activities for the user, `ORDER BY start_date ASC` at the database level. In Python, group by calendar date (`start_date.date()`), sum `distance_meters` per day, then accumulate a running total. Each calendar date appears at most once in `daily_series` (multiple runs on the same day are merged into one point). Sorting is guaranteed by the DB — no Python sort needed, no crash risk.

## Frontend Changes

### `frontend/src/api/client.ts`

Add two typed functions:

- `getPersonalDashboard() -> PersonalDashboard | null` — returns `null` on 404 (`not_synced`), throws on other errors
- `putGoal(km: number) -> GoalResponse`

Types:

```ts
interface DailyDistancePoint {
  date: string        // YYYY-MM-DD
  cumulative_km: number
}

interface PersonalDashboard {
  goal_km: number
  distance_to_date_km: number
  progress_pct: number
  on_pace: boolean
  expected_pct: number
  last_sync_completed_at: string
  daily_series: DailyDistancePoint[]
}

interface GoalResponse {
  yearly_running_goal_km: number
}
```

### `frontend/src/pages/DashboardPage.tsx`

On mount, fetch `getPersonalDashboard()`. After a successful sync, re-fetch. After a successful goal save, re-fetch.

**Layout (top to bottom):**

1. Existing sync card (unchanged)
2. Stats row — three tiles: Total km, Complete %, Pace indicator
3. Pace chart card — Recharts `AreaChart`
4. Goal edit card — number input + save button

**Stats row tiles:**

- "Total km": `distance_to_date_km` formatted to 1 decimal, JetBrains Mono 20px, `--accent` color
- "Complete": `progress_pct` formatted to 1 decimal + "%", same treatment
- "Pace": text badge — "On pace" (`--success` text, `--success-dim` bg) or "Behind pace" (`--warning` text, `--warning-dim` bg)

**Pace chart (`AreaChart` from Recharts):**

- Actual distance series: `daily_series` data, blue area fill (`rgba(75,140,247,0.08)`) + line (`--accent`)
- Goal pace line: computed on the frontend — a straight line from `{date: Jan 1, cumulative_km: 0}` to `{date: Dec 31, cumulative_km: goal_km}`, rendered as a dashed `Line` in `--text-3`
- X-axis: month labels (Jan–Dec), `--text-3`, 11px JetBrains Mono
- Y-axis: km values, same treatment
- Grid lines: `--border`
- Tooltip: `--surface-2` background, `--border` border, 8px radius
- Chart occupies full card width; responsive via `ResponsiveContainer`

**Goal edit card:**

- Number input pre-filled with current `goal_km`
- "Save" button (primary)
- Frontend validation: 1–100,000 km; error shown inline before submit
- Saving state: button shows spinner + disabled
- Save error: inline error message inside the card
- On success: re-fetch `getPersonalDashboard()`

## Empty States

| Condition | Behaviour |
|---|---|
| Not synced (404 `not_synced`) | Single card: "No running activities yet — sync your data to get started." No stats, no chart, no goal edit. |
| Synced, zero runs (200, `daily_series: []`) | Stats show zeros; chart card shows "No runs recorded this year yet." in place of the chart; goal edit renders normally. |
| API error on load | Inline error message with a retry button; no crash. |

Goal save errors are shown inline in the goal edit card. Sync errors are handled by the existing sync card.

## Dependency

Recharts must be installed: `npm install recharts`.

Recharts color values are passed as inline props (strings). CSS variables (`var(--accent)`) do not work in SVG `stroke`/`fill` props in all browsers — read them via `getComputedStyle(document.documentElement)` at render time or use hardcoded design-token values with dark/light mode awareness via the existing `data-theme` attribute.
